25// NOTE: GRAPH DATA STRUCTURES/FORMATS:
26// in the long term, ideally we should use boost graph data formats
27// (i.e., adjacency_list) or some other type for ALL graphs in tket.
28//
29// Assuming that we do like boost::adjacency_list, which has a lot
30// of extra template parameters, it seems there are 4 options.
31// In increasing order of work/code complexity:
32//
33// (i): do nothing (the short term option!)
34//
35// (ii): create boost::... -> {our data structures} conversion routines,
36// and continue to use our routines unchanged.
37// (e.g., brute force colouring is the bottleneck, not conversion).
38//
39// (iii): create {general boost::...} -> {special restricted boost::...}
40// conversion routines, and change our algorithms to use the
41// restricted boost objects. (With all template parameters fixed,
42// and all irrelevant extra vertex data stripped, so at least
43// we can be sure of efficiency in implementation).
44//
45// (iv): change all routines to use general boost::... directly,
46// with templates everywhere, and POSSIBLY unavoidable inefficiencies,
47// because the boost graph stuff is very general.
48//
49// Another option is to go Java-style and use interfaces/virtual functions
50// everywhere instead of templates.
51
