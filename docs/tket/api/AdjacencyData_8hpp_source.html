26 // NOTE: GRAPH DATA STRUCTURES/FORMATS:
27 // in the long term, ideally we should use boost graph data formats
28 // (i.e., adjacency_list) or some other type for ALL graphs in tket.
29 //
30 // Assuming that we do like boost::adjacency_list, which has a lot
31 // of extra template parameters, it seems there are 4 options.
32 // In increasing order of work/code complexity:
33 //
34 // (i): do nothing (the short term option!)
35 //
36 // (ii): create boost::... -> {our data structures} conversion routines,
37 // and continue to use our routines unchanged.
38 // (e.g., brute force colouring is the bottleneck, not conversion).
39 //
40 // (iii): create {general boost::...} -> {special restricted boost::...}
41 // conversion routines, and change our algorithms to use the
42 // restricted boost objects. (With all template parameters fixed,
43 // and all irrelevant extra vertex data stripped, so at least
44 // we can be sure of efficiency in implementation).
45 //
46 // (iv): change all routines to use general boost::... directly,
47 // with templates everywhere, and POSSIBLY unavoidable inefficiencies,
48 // because the boost graph stuff is very general.
49 //
50 // Another option is to go Java-style and use interfaces/virtual functions
51 // everywhere instead of templates.
