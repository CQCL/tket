27// TODO: if we only allow 64 colours
28// (which follows if, e.g., we only have <= 64 vertices),
29// then it can be done faster with bit operations.
30// 64^64 > 10^115 is ridiculously big, so it's not very likely
31// we'll get a real graph, which we have any realistic
32// hope of colouring well, needing more than 64 colours.
33// For C colours, a potential speedup of roughly a factor C.
34//
35// If we are going to find the minimum possible number of colours
36// to colour a graph, it is exponential time in the worst case. However,
37// it still might be possible to speed it up substantially from brute force,
38// if we colour the vertices in a particular order.
39// As we descend along the sequence of colourings, we will hit
40// colouring inconsistencies and have to backtrack.
41// But if we have selected a good order, we might hit inconsistencies
42// quite early on average, and hence prune a lot of the tree
43// of all possibilities, so we hopefully examine only a fraction
44// of all candidates, yet still guarantee to find an optimal colouring.
45//
46// (In fact, this clearly is happening, since we are able to colour,
47// apparently optimally, some fairly dense graphs on 30 vertices in << 1 second
48// with 3 colours. A trivial brute force colouring would need 3^30 ~ 2*10^14
49// operations, so it is many orders of magnitude faster than that).
50//
